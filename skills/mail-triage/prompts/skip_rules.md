# Skip Rules — 3層判定ルール

> ドラフト生成スキップ判定。受信メールを「返信ドラフトを作るに値するか」で3層に通す。
> 通過 = ドラフト生成、スキップ = 人間も触らない。

---

## 設計の起源（D-019 反映）

> 機械的に全部下書き作ると、不要返信・越境誤爆・上から目線が混入する。
> **3層フィルタを通したものだけがドラフト候補になる**。

| 層 | 名前 | 役割 | 発火タイミング |
|---|---|---|---|
| L1 | 越境誤爆 | 自分のスレに他人が割り込まない | ドラフト生成**前** |
| L2 | 沈黙 | 既決着スレで不要返信しない | ドラフト生成**前** |
| L3 | 上から目線 | 装飾語・依頼トーンを警告 | ドラフト生成**後（post-check）** |

L1/L2 は skip = 処理打ち切り。L3 は警告のみ（人間が修正判断）。

---

## L1: 越境誤爆

**ロジック**: メールスレッドに `ball_with`（次にアクションすべき人）属性を持たせる。
ball_with が**自分以外の協働者**を指す場合、自分が割り込まない。

```
checkOverreach(email):
  ball = email.ball_with
  if ball が COLLABORATORS リスト内の誰か:
    → skip "overreach: ball_with=XXX"

  if 最新送信者が自分でない AND ball に「自分」が含まれない:
    → skip "overreach: latest_sender=XXX, ball=YYY"

  else:
    → 通過
```

### COLLABORATORS リストの定義

外部化推奨（D-026）。ハードコードしない。

```yaml
# config/collaborator_map.yaml の例
collaborators:
  - name: 営業担当 A
    aliases: [営業A, A-san]
    role: 商談主担当
  - name: 経理担当 B
    aliases: [経理B, B-san]
    role: 請求・支払
  - name: 法務担当 C
    aliases: [法務C, C-san]
    role: 契約レビュー
  # ... 自組織で追加
```

スキル本体はこのファイルを読み込んで `COLLABORATORS` 配列を構築。
**人事異動で書き換えやすい構造にする**。コードに名前を埋め込まない。

### 判定例

| ball_with | latest_sender | 判定 |
|---|---|---|
| 営業担当 A | 営業担当 A | ✅ skip（A のスレに自分が割り込まない） |
| 自分 | 取引先 X | 通過（自分宛・要対応） |
| `?` / `none` / `-` | 取引先 X | 通過（ball 不明・要判定） |
| 経理担当 B | 自分 | ✅ skip（自分が B にボール渡した。B 待ち） |

## L2: 沈黙

**ロジック**: 既に「ありがとう」「承知」で締まったスレに、追加返信しない。

```
checkSilence(email, threadHistory):
  ball = email.ball_with
  最新本文 = email.body_preview

  if ball が「自分」 AND 最新送信者が自分:
    if 最新本文に /ありがとう|承知|了解/ AND 80字以下:
      → skip "silence: closed-thank-you-loop"

  if email.ball_type == "self-awaiting-reply"  AND 最新送信者が自分:
    → skip "silence: already-replied-awaiting"

  else:
    → 通過
```

### 判定例

| 状況 | 判定 |
|---|---|
| 自分が「ありがとうございます。承知しました。」を最後に送信した | ✅ skip（決着済） |
| 自分が「お時間ありがとうございます。〜の件、ご連絡お待ちしてます」を送信（80字超 + 依頼） | 通過（待ちが残ってる = 観察対象） |
| 取引先が「ありがとうございます」を返してきた直後 | 通過（追加メッセージありかもしれない・要判定） |

**80字閾値**は経験則。短いほど「終了の挨拶」、長いと「依頼が混ざってる」可能性が上がる。

## L3: 上から目線（post-check）

**ロジック**: ドラフト生成**後**の警告層。skip ではなく「人間が修正検討」のフラグ立て。

```
checkCondescension(body):
  hasThanks = /ありがとう|感謝|誠にありがたく/.test(body)
  hasRequest = /ご一報|お知らせください|ご連絡ください|お送りください|お願いいたします後|お待ちしております/.test(body)

  if hasThanks AND /ご一報|お知らせください|お送りください/.test(body):
    → 警告 mid: "thank+request 同時出現"

  decorations = ['誠に', '謹んで', '差し支えなければ', '恐縮ながら']
  if body に decorations のいずれか:
    → 警告 low: "装飾語検出"

  else:
    → 通過
```

### なぜ「上から目線」と呼ぶか

「ありがとうございます」と言いつつ即「ご連絡ください」とつなげると、**感謝してるフリで催促してる**ように見える。
人間は読んだ瞬間に違和感を覚える。LLM が生成しがちな典型パターン。

### 装飾語が嫌われる理由

「誠に」「謹んで」「差し支えなければ」「恐縮ながら」は**形式美**。
社内メール・近しい取引先には冗長で、距離感を逆に作ってしまう。
カジュアルに「お時間あるとき教えてください」で十分。

## 統合判定 `shouldSkip(email)`

```
skip = false
reasons = []

if checkOverreach(email).skip:
  skip = true; reasons.push(L1 reason)

if checkSilence(email).skip:
  skip = true; reasons.push(L2 reason)

return { skip, reasons }
```

L3 の警告は別フロー（draft 生成後に通す）。

## PRECEDENCE（複数該当時の優先順位）

複数の層に同時にひっかかった時は、**早い方を優先**する。
判定が安定し、AI の気分でブレなくなる。

```
L1 越境誤爆  >  L2 沈黙  >  L3 上から目線（警告のみ）
```

### 適用ルール

1. L1 が skip と判定したら → **その時点で打ち切り**、L2/L3 を見ない
2. L1 が通過 → L2 を見る。L2 が skip ならそこで打ち切り
3. L1/L2 ともに通過 → draft を生成し、その後 L3 の post-check
4. L3 は skip ではなく**警告**。人間が修正するかどうか判断

### なぜこの順か

- **L1 が一番上**: 越境（他人のスレに割り込み）は**送るべきでない**判断。最も致命的。
- **L2 が次**: 沈黙（既決着スレへの追加返信）は**送ると鬱陶しい**判断。中程度。
- **L3 が最後**: 上から目線は**書き方の問題**。送ること自体は許容、文面を直せば済む。

### 例

```
受信メール: ball_with=「営業担当 A」、本文に「ありがとうございます」+ 短文
```

このメールは:
- L1 で skip 判定（越境＝A のスレに割り込まない）
- L2 もたぶん skip 判定（既決着）
- L3 は draft 生成しないので評価されない

→ L1 で打ち切るので、L2/L3 を計算するコストも省ける。

## カスタマイズポイント

| 箇所 | 何を書き換えるか | 例 |
|---|---|---|
| `COLLABORATORS` | 自組織の協働者リスト | 上司・同僚・主要取引先 |
| 沈黙の80字閾値 | 業務スタイルに合わせる | 短文文化なら60字、丁寧文化なら120字 |
| L3 の依頼語句リスト | 自分が違和感ある語の追加 | 「ご教示」「ご査収」を追加するか |
| L3 の装飾語リスト | 自分のNG語 | 「お忙しいところ恐縮ですが」を追加するか |

---

## このルールの限界

- **新規取引先で `ball_with` 属性が未設定**: L1 が空振り → 通過してしまう。本文判定で補う必要
- **複数言語混在メール**: 正規表現は日本語前提。英語混じりは別途対応
- **VIP からの依頼**: L1 で skip されることがある（VIP のスレに割り込まない判定）→ VIP リストで例外復元する設計が必要
