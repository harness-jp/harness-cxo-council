## 数字の根拠（採点ループの実測値）

直近の P-01 ベースライン採点（5月時点）の数字:

| 指標 | 値 | 意味 |
|---|---|---|
| **ok 率** | **82%** | 採点 ▲（そのまま送れる）の割合。10件中8件は人間が削るだけで送れる |
| **skip 妥当率** | **100%** | skip_rules で skip 判定したものが全て正解（5/5）。誤 skip ゼロ |
| **構造バグ修正** | **3 つ** | 採点で見つかった「LLM が繰り返す型バグ」を skip_rules.mjs に反映済 |

### 採点フロー

```
受信メール
 ↓
[PB 分類] 8 PB のいずれかに割当
 ↓
[skip_rules 3層] L1越境 / L2沈黙 / L3上から目線 を通過
 ↓
[draft 生成] PB 別テンプレで埋める
 ↓
[scoring UI (HTML)]    ← 人間がブラウザで開いて採点
   - ▲ = そのまま送れる
   - △ = 微修正で送れる
   - ✗ = 大幅修正 or 不要
 ↓
[JSON 蓄積]            ← data/baselines/p{XX}_baseline_{date}.json
 ↓
[few-shot 注入]         ← 次回の draft 生成プロンプトに「過去の良い例 N件」を注入
 ↓
精度が上がる（見かけ上の「学習」）
```

**ポイント**: モデルは学習していない。**外部メモリ（JSON）が厚くなることで retrieval の質が上がる**だけ。
これが「学習を諦めて引用を強要する」哲学の具体例。

## 構造バグ3つ（実例）

採点で見つかった「LLM が繰り返す型のミス」と修正:

### バグ1: 感謝後の依頼（mail_tone_no_request_after_thanks）

**症状**:
```
お忙しいところありがとうございます。
〜の件、ご連絡ください。
```

**LLM の癖**: 丁寧に始めようとして「ありがとうございます」を冒頭に置く。
直後に依頼を続けると、相手から見ると**感謝のフリで催促**してる印象になる。

**修正**: skip_rules.mjs の L3 `checkCondescension` で「感謝＋依頼」同時出現を警告検出。
人間が削る → 「お忙しいところすみません」または感謝を別段落に分離。

### バグ2: 沈黙判定失敗（silence_classification_failure）

**症状**:
スレッドが「ありがとうございます」「承知しました」で終わってるのに、
LLM が「念のため確認の返信を作りますね」と勝手にドラフトを生成。

**LLM の癖**: 「丁寧に1往復追加する方が安全」と判断しがち。実際は不要。

**修正**: skip_rules.mjs の L2 `checkSilence` で「自分が感謝で締めた + 80字以下 → skip」を機械判定。

### バグ3: 越境誤爆（overreach）

**症状**:
他の人が主担当のスレッド（`ball_with: 営業担当 A`）に、自分の draft が生成される。
「私の方からも一言」のような形で割り込み。

**LLM の癖**: メールに自分が CC されてると「自分も発言しなきゃ」と判定する。

**修正**: skip_rules.mjs の L1 `checkOverreach` で `ball_with` を見て、自分以外の協働者なら skip。

## 失敗事例: 4重複合原因の48時間滞留

### 何が起きたか（汎化）

ある重要取引先からの返信が、**48時間以上**自分の処理から漏れた。
発覚時には商談が止まりかけていた。

### 原因（4重複合）

単一原因じゃなかった。**4つが同時に効いて**滞留が起きた:

| # | 原因 | 仕組み |
|---|---|---|
| 1 | **TZバグ** | 日本時間/UTC 取り違えで「昨日」の送信が「今日扱い」になり、未返信追跡から漏れた |
| 2 | **モデル動作の特性** | 過去スレッド = 既知 = 優先度低、と LLM が判定。新規メールばかり surface |
| 3 | **Re:/Fw: の軽視** | 件名に Re: が付くと「進行中の継続」扱い。実は滞留中のリマインドだったのに埋もれた |
| 4 | **Outlook warning による停止** | 警告ダイアログで処理が止まって、そのスレッドだけ取得失敗してた |

**1つでも起きていれば自力で立ち直れた**。4つ同時で穴が空いた。

### 対処（全部入り）

1. **重要取引先 domain ホワイトリスト**を事前定義、最優先深掘り
2. **Re:/Fw: でも新規扱い**（過去スレッド軽視を防ぐ）
3. **Outlook warning 無視ルール**（警告で止まらない）
4. **3日チェックモード**（短期再確認サイクル）

「単一原因として事故処理しない」のが**今後の運用方針**。

### 教訓

事故が起きたら、**4重5重の同時発生を疑え**。
1つの fix だけでは穴が残る。**全部入り対処**が現実解。

## 「採点 → 改善」サイクルの回し方

### 1日の運用

```
朝:
  メールチェックトリガー → トリアージ + draft 生成
  10分で確認 → 送信ボタン or 修正

夕方（任意）:
  scoring_ui.html を開いて、その日の draft を採点
  ▲ / △ / ✗ を付ける（5-10件 / 10分）
  JSON に追記

週次:
  ▲率（ok率）が下がっているなら原因特定
  構造バグなら skip_rules / templates を更新
  個別ケースなら few-shot に追加
```

### 「精度を上げる」とは何か

精度の正体を分解する:

| 軸 | 何を意味するか | 北極星候補 |
|---|---|---|
| 意図正解率 | 受諾/拒否/質問返し の方向性が合ってる | △ ある程度自明 |
| **そのまま送れる率（ok率）** | 編集ゼロで送信ボタン押せる | ⭐ **これが現場の指標** |
| トーン一致率 | 自分の文体・敬語レベルが再現されてる | ◯ ok率に内包される |
| 引用妥当性 | 引用された過去メールが本当に類似 | ◯ 抜き打ちチェック |
| ハルシネーション率 | 嘘事実の混入率 | △ Codex Gate で抑える |

**「そのまま送れる率」を北極星にする**。82% → 90% を目指す、のような目標で運用する。
編集が必要 = 失敗ではなく learning_signal、と扱う。

## やってないこと（あえて）

ユーザー個人の好みを学習する**AIループ**は実装していない。

| やったこと | やってないこと |
|---|---|
| ✅ 採点結果を JSON に蓄積 | ❌ プロンプト再注入の自動パイプ |
| ✅ few-shot に過去の良い例を入れる | ❌ ユーザー編集差分の自動学習 |
| ✅ 構造バグを skip_rules に反映（人手） | ❌ モデル fine-tune |

理由:
1. 「人手で削れば OK ベース」の運用方針（ドラフトが多めなら人間が削る方が安全）
2. 学習を諦めて引用を強要する哲学（[prompts/citation_enforcement.md](../../../prompts/citation_enforcement.md) と整合）
3. 自動学習はやって失敗した過去あり（[harness-cxo-council README](../../../README.md) の「中心思想」参照）
